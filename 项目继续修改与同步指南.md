# 成她100 · 继续修改与同步指南

最后更新：2026-06-09

这份说明给后续接手的 AI 或开发者使用。重点是：不要找错文件夹，不要直接覆盖现在已经修好的内容，先查历史，再小范围修改，最后用固定脚本同步上线。

---

## ★ 关于"做完"的定义

**每一次修改完成的标志 =推送成功 + Vercel 部署 Ready**

完整流程：
1. 修改代码
2. `npm test` + `npm run lint` + `npm run build` 全部通过
3. `./scripts/sync-to-github.sh "更新说明"` 执行成功
4. `vercel inspect chengta-100.vercel.app --scope hellary` 显示 **status: ● Ready**

只有以上四步全部完成，才算"做完"。仅推送到 GitHub 但 Vercel 还未部署完成不算结束。

## 一、唯一修改目录

所有代码和内容修改都在这里做：

```bash
cd /Volumes/PS2000/成她
```

不要在这些地方直接改业务代码：

```bash
/Users/macbook/Documents/GitHub/-100
```

这个目录是同步目标仓库，用来提交到 GitHub 和触发 Vercel。它可以用来查干净的 Git 历史，但不要把它当主工作区改。因为每次运行同步脚本时，PS2000 会覆盖同步目标。

也不要新建其它同名项目文件夹，例如 Verso、桌面、下载目录里的临时副本。之前混乱就是因为多个目录同时存在，导致改了一个地方，线上部署的是另一个地方。

## 二、固定同步流程

修改完成后，在 PS2000 项目根目录执行：

```bash
cd /Volumes/PS2000/成她
npm test
npm run lint
npm run build
./scripts/sync-to-github.sh "更新说明"
```

示例：

```bash
./scripts/sync-to-github.sh "修复知识库渐隐和测评入口"
```

这个脚本会做三件事：

1. 把 `/Volumes/PS2000/成她` 同步到 `/Users/macbook/Documents/GitHub/-100`
2. 在 `/Users/macbook/Documents/GitHub/-100` 里执行 `git add .`、`git commit`
3. 推送到 GitHub `main`，然后 Vercel 自动生产部署

脚本位置：

```bash
/Volumes/PS2000/成她/scripts/sync-to-github.sh
```

脚本已经排除了这些本地残留：

```bash

```

## 四、查找“做过但被覆盖”的内容

查历史建议在同步目标仓库执行，因为这里的 Git 历史更干净：

```bash
cd /Users/macbook/Documents/GitHub/-100
```

查看最近提交：

```bash
git log --oneline --decorate --graph --all -30
```

查看某个文件的修改历史：

```bash
git log --oneline -- src/components/AssessmentResultClient.tsx
git log --oneline -- src/app/knowledge/page.tsx
git log --oneline -- src/components/HomeDashboard.tsx
```

看某个旧提交里的文件内容：

```bash
git show <commit>:src/components/AssessmentResultClient.tsx
git show <commit>:src/app/knowledge/page.tsx
```

对比旧提交和当前版本：

```bash
git diff <commit>..HEAD -- src/components/AssessmentResultClient.tsx
git diff <commit>..HEAD -- src/app/knowledge/page.tsx
```

按关键词找“某句话/某功能”是哪次加的或删的：

```bash
git log -S "为什么从这里开始" --all -- src/components/AssessmentResultClient.tsx
git log -S "推荐起点" --all -- src/components/AssessmentResultClient.tsx
git log -S "knowledge-grid" --all -- src/app/knowledge/page.tsx src/app/globals.css
```

按关键词搜当前工作区：

```bash
cd /Volumes/PS2000/成她
rg -n "为什么从这里开始|推荐起点|查看测评报告|Day 7|神秘卡|工具卡" src
```

找本地内容源文档：

```bash
rg --files | rg "Day|IFS|提问|工具箱|测评|知识库"
```

恢复历史内容时不要直接 `git checkout <commit> -- 文件` 覆盖当前文件。正确做法是：

1. 先 `git show <commit>:文件路径` 看旧内容
2. 判断旧内容是否仍然适合当前版本
3. 只把需要的片段手动合并到 `/Volumes/PS2000/成她`
4. 跑测试和构建
5. 用同步脚本提交

## 八、不要乱改的边界

不要为了修一个页面顺手重构全站结构。

不要把 Day 1-7 正文改回占位文案。正式正文来自：

```text
成她-Day1-7.md
src/lib/day-document.ts
```

不要把 Home 改回“新用户只能联系开通会员”。当前逻辑是：

```text
新用户进入 Home 后弹窗提示先做测评；
可以关闭弹窗继续浏览知识库；
知识库可浏览，AI/会员能力另行限制。
```

不要把“查看测评报告”重新放回 Home 右上角。当前要求是右上角只保留“我的匣子”。

不要把报告页的 Day 推荐入口做成多个跳转。当前要求是：

```text
只保留一个主要入口，直接跳到推荐 Day。
三个小标签可以有 Tooltip，但不要变成多个重复入口。
```

不要直接覆盖 `src/components/MysteryCard.tsx` 的翻牌逻辑。当前要求是：

```text
卡片可翻牌；
背面长文本可点击展开；
卡册按 25 张工具卡总数展示，不按 Day 排序，不显示 Day 序号。
```
